# 用户分配记录查询工作台 - 大圣部署说明

## 项目信息

项目类型：Next.js

项目目录：

`电销分配例子查询工作台/project-download`

启动命令：

```bash
pnpm install
pnpm run build
pnpm run start
```

默认端口：

`8000`

## 真实数据查询方式

工作台后端接口会调用大数据 MCP 的 adhoc 查询能力。

需要部署环境支持服务端调用大数据 MCP，并配置以下环境变量：

```bash
ALLOCATION_QUERY_PROVIDER=bigdata-mcp
ALLOCATION_BIGDATA_MCP_URL=大数据MCP调用地址
ALLOCATION_BIGDATA_MCP_TOKEN=大数据MCP Token
ALLOCATION_BIGDATA_MCP_TIMEOUT_SECONDS=60
ALLOCATION_USE_MOCK=false
```

注意：

- Token 只在服务端使用，不要配置成 `NEXT_PUBLIC_`。
- 如果不配置 `ALLOCATION_QUERY_PROVIDER=bigdata-mcp`，项目会回退到 mock 或通用 SQL API。

## MCP 调用协议

当前代码按下面的 MCP HTTP wrapper 形式调用：

```json
{
  "toolName": "adhoc_submitQuery",
  "arguments": {
    "sql": "SELECT ...",
    "engine": "Spark",
    "catalog": "hive_f04",
    "database": "dw_ads",
    "timeoutSeconds": 60
  }
}
```

如果大圣侧 MCP HTTP wrapper 的参数格式不同，需要对应调整：

`app/api/allocation/data-source.ts`

中的 `executeBigDataMcp` 方法。

## 日期口径

BPO 的原始 `dt` 比业务展示分配日期早 1 天。

所以 BPO 查询里必须使用：

```sql
date_add(to_date(dt), 1) AS dt
```

TMK 直接使用原始 `dt`。

CC 直接使用原始 `dt`。

## 接口

查询 UID：

`POST /api/allocation/query`

导出明细：

`POST /api/allocation/export`

导出限制：

- 支持单日或时间段。
- 时间段最多 14 天。

## 每日缓存（推荐开启）

先执行一次数据库初始化，创建缓存表：

```bash
pnpm run db:init
```

升级“实际分配/拨打/接通”版本时，保留原缓存表并执行：

```bash
pnpm run db:migrate:assignment-contact
```

部署环境增加：

```bash
ALLOCATION_CACHE_ENABLED=true
ALLOCATION_CACHE_REFRESH_TOKEN=随机长字符串
ALLOCATION_CACHE_RETENTION_DAYS=90
ALLOCATION_CACHE_QUERY_TIMEOUT_SECONDS=600
```

每天上午在数仓产出完成后，由大圣定时任务调用。新版本需要刷新今天及前两天，
让今天进入缓存，并回补晚发生的拨打、接通和转人工分配：

```bash
ALLOCATION_CACHE_ROLLING_DAYS=3 pnpm run cache:refresh:rolling
```

旧的 `pnpm run cache:refresh` 默认只刷新昨天，因此不会把今天加入缓存。
不要再一次请求刷新三个渠道，使用脚本逐渠道执行，避免HTTP超时：

```bash
bash scripts/refresh-cache-channels.sh
```

脚本依次刷新 BPO、TMK、CC；每个渠道最长等待900秒，任一渠道失败会返回非0退出码，
供11:00失败重试任务判断。

首次上线可分批补历史数据，单次最多 31 天：

```bash
pnpm run cache:backfill -- 2026-04-18 2026-07-16
```

脚本会自动按最多30天一批，并按 BPO、TMK、CC 逐渠道执行。也可以手工调用：

```bash
curl -X POST http://服务地址/api/allocation/cache/refresh \
  -H "Authorization: Bearer $ALLOCATION_CACHE_REFRESH_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"startDate":"2026-06-01","endDate":"2026-06-30"}'
```

工作台查询时会优先读取缓存；日期尚未刷新时自动回退实时数仓查询。

说明：31 天只是单次刷新任务的保护上限。缓存按
`ALLOCATION_CACHE_RETENTION_DAYS` 动态清理，当前默认保留90天。

## 实际分配与触达口径

- 商分入池：原 BPO / TMK / CC 名单表，仅表示进入候选池。
- TMK 实际分配与触达：`dw_conan_ads.ads_conan_tmk_hunt_lead_day_di`，限定 `feed_lead_source='internal'`。
- BPO 实际分配与触达：同一 Hunter 日宽表，限定 `feed_lead_source='side'`。
- Hunter 正式下发：`is_in_feed=1`；BPO / TMK 的实际下发日比 `feed_source_dt` 晚一天。
- Hunter 转人工：`is_manual_assigned=1`，仅表示拨通后进入真人商机，销售为 `assign_ldap`。
- Hunter 拨打/接通：`is_dialed=1` / `is_connected=1`。
- CC 实际分配：通用商机表 `leadtype='20'`，接通取外呼记录正常接通状态。
- CC 名单日到实际分配日通常为 T+1，代码按 `商分池 dt + 1 = 商机开始日` 关联。
- 不得再把“进入商分池”展示成“已经分配销售”。

Hunter 日宽表从 `2026-06-01` 开始有分区；更早日期只能保留商分入池信息，
实际分配与触达字段会显示为否/空。
